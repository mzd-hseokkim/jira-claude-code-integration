02

핵심 기능

🗂️
Multi-Worktree 병렬 개발
/jira-task init 5 하나로 할당된 태스크 N개의 git worktree를 일괄 생성합니다. 각 태스크가 독립된 디렉토리와 브랜치에서 병렬 진행되며, 컨텍스트 전환 없이 여러 이슈를 동시에 다룰 수 있습니다.
📄
문서 자동 생성 & Jira 연동
기획(plan.md), 설계(design.md), 테스트 리포트, 코드 리뷰 결과가 자동 생성되어 Jira 이슈에 첨부파일과 코멘트로 즉시 게시됩니다. 별도 복사·붙여넣기 없이 개발 산출물이 Jira에 축적됩니다.
🔄
상태 전이 자동화
start → In Progress, merge → In Review, done → Done까지 Jira 상태 전이가 커맨드에 연동됩니다. 개발자가 Jira를 직접 열지 않아도 이슈 상태가 자동으로 갱신됩니다.
🔍
설계-구현 Gap 분석
/jira-task review는 설계 문서와 실제 구현 코드를 비교해 Gap을 자동 분석하고, 코드 품질까지 검사해 Jira 코멘트로 게시합니다.
📊
Progress 자동 추적
.jira-context.json에 완료 단계가 기록되어, 다음 실행 시 현재 진행 상황이 자동으로 표시됩니다. 세션이 끊겨도 어디까지 했는지 항상 파악할 수 있습니다.
🪝
Session Hooks
세션 시작 시 활성 태스크 정보를 자동 표시하고, 세션 종료 시 진행 중인 태스크가 있으면 Jira 동기화 리마인더를 표시합니다. 작업 컨텍스트가 세션 간에 자동 복원됩니다.
03

Multi-Worktree 구조와 PR 병합 전략

모든 worktree는 동일한 base 브랜치(main/develop)에서 생성됩니다. 서로 다른 태스크가 같은 파일을 수정하면 병합 시 충돌이 발생할 수 있어, 설계 단계에서 미리 전략을 정하는 것이 핵심입니다.

Worktree 구조
workspace/
├── your-project/ # main 브랜치
└── your-project_worktree/
      ├── PROJ-101/ # feature/PROJ-101
      ├── PROJ-102/ # feature/PROJ-102
      └── PROJ-103/ # feature/PROJ-103
충돌 발생 원리
모든 브랜치가 동일 시점의 main에서 분기됩니다.
PROJ-101이 main에 병합되면, PROJ-102는 그 변경을 모르는 상태가 됩니다.

충돌 여부는 파일 중복 여부로만 결정됩니다.
설계 단계에서 git diff --name-only main feature/PROJ-XXX 로 미리 확인하는 것이 핵심입니다.
전략 1
순차 Rebase-and-Merge
병합 순서를 사전에 정하고, 앞 태스크가 main에 합쳐진 뒤 뒷 태스크를 rebase해서 PR을 올립니다. 충돌을 작은 단위로 나눠 해결합니다.
적합한 경우: 태스크 간 의존 관계 있음, 같은 파일 수정
전략 2
통합 브랜치
별도 통합 브랜치(feature/sprint-42)를 만들고, 각 태스크 PR을 그곳으로 먼저 합친 뒤 스프린트 종료 시 main으로 PR 1개만 올립니다.
적합한 경우: 여러 태스크를 함께 릴리즈, 개별 배포 불가
전략 3
사전 파일 영역 분리
설계 단계에서 수정 파일이 다른 진행 태스크와 겹치지 않도록 설계합니다. 겹치는 파일이 없으면 어떤 순서로 병합해도 충돌이 없습니다.
적합한 경우: 독립적인 기능 영역, 가장 단순한 방법
의사결정 흐름 파일 중복 없음 → 전략 3  |  중복 있음 + 각자 배포 가능 → 전략 1  |  중복 있음 + 함께 릴리즈 → 전략 2
개발 기간 중에는 매일 또는 세션 시작 시 git fetch + rebase origin/main으로 충돌을 조금씩 해소하는 것이 최선
04

설치 & 셋업

Step 1
플러그인 설치
Claude Code 마켓플레이스에서 플러그인을 설치합니다.
claude plugin marketplace add mzd-hseokkim/jira-claude-code-integration claude plugin install jira-integration@jira-claude-code-integration
Step 2
Jira API Token 생성
Atlassian 계정 설정에서 API Token을 생성합니다. 토큰은 생성 시 한 번만 표시됩니다.
id.atlassian.com → Security → API tokens
Step 3
MCP 서버 등록
mcp-atlassian을 Claude Code에 등록합니다. uvx로 실행해 별도 설치 불필요. 환경변수 4개 설정 필요.
claude mcp add atlassian -e JIRA_URL=... -e JIRA_USERNAME=... -e JIRA_API_TOKEN=... -- uvx mcp-atlassian
Step 4
연결 확인 & 시작
Claude Code 실행 후 연결을 확인하고 할당 태스크를 가져옵니다.
> /jira > /jira-task init 5
주의 API 토큰이 저장되는 .claude/settings.local.json은 반드시 .gitignore에 추가할 것. 기존 bkit 플러그인이 설치되어 있다면 먼저 삭제 필요 (스킬/훅 충돌).

로드맵

Git Hosting 확장
  • Bitbucket Cloud PR 생성 지원 (현재 GitHub gh CLI만 지원)
  • GitLab MR 생성 지원 (glab CLI)
  • Git remote URL 자동 감지 → 플랫폼 분기 처리
Jira 확장
  • Jira Server/Data Center 지원 (Personal Access Token 모드)
  • 커스텀 워크플로우 매핑 (회사별 transition 이름 설정)
  • Sub-task 자동 생성 (설계 Task Breakdown → Jira sub-task)
  • Time Tracking 자동 기록
테스트 & 품질
  • Playwright 테스트 자동 생성 고도화 (acceptance criteria 기반)
  • 테스트 커버리지 Jira 코멘트 연동
  • CI/CD 결과 Jira 자동 게시 (GitHub Actions / Jenkins)
알림 & 사용성
  • Slack / Teams 알림 연동 (PR 생성, 완료 시)
  • 대화형 설정 마법사 (/jira setup)
  • 전체 태스크 현황 대시보드 뷰
  • npm 패키지화 → 원클릭 설치
